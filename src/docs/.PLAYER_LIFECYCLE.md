# Player Character Lifecycle

Documents the full arc of a `Player` object from the moment someone joins a game to the end of
the epilogue. Covers both Dungeon Mode and Town Mode where they differ.

---

## 1 — Joining & Initial State

A player joins by entering a game code in the player-client. On creation a `Player` is initialized
with:

- `authorId` — random UUID, the player's stable identity throughout the game
- `userName` — the name they typed
- `joinedAt` — timestamp used for all phase-assignment offset calculations (sort order = join order)
- Seven base stats, all set to **4**: `strength`, `dexterity`, `charisma`, `intellect`, `wealth`,
  `magic`, `favor`
- Empty `traits: []`, `titles: []`
- `playerClass: null` (until they select one)

---

## 2 — Character Creation (INIT)

Shown via the `player` component in the player-client. Two steps completed before the game begins:

### Step 1 — Class Selection

`GET /player/classes` returns all four `PlayerClassOption` values. The player picks one:

| Class | What it unlocks |
|---|---|
| **Bard** | Can spread outcomes to all players (`ALL_PLAYERS`) |
| **Herald** | Can give titles to other players (`TITLE`) |
| **Scribe** | Can give traits to other players (`TRAIT`) |
| **Fabulist** | Can convert encounters into companion traits (`COMPANION`) |

Stored on `Player.playerClass`. Gates which repercussion types appear in the `repercussion-toggle`
component during writing phases. Class does not change over the course of the game.

### Step 2 — Trait Selection

Player picks one starting trait from the available `Trait[]` pool. Appended to `Player.traits`
as `TraitType.STANDARD`.

### displayName

`Player.getDisplayName()` is computed, not stored: `userName` + any `TraitType.TITLE` traits
joined with commas. Example: "Morwen, The Unbroken, Voice of the Vale".

---

## 3 — PREAMBLE

Static text establishes the narrative stakes for all players — where they are, what is coming,
and what the journey will demand of them. No player input is required; this is a read-only
information phase shown on the shared game-client screen.

---

## 4 — Location Selection (LOCATION_VOTING, locationVoting mode)

Players vote on where they will travel first. The candidates come from `AdventureMap.locations`
(the setting's defined places — e.g., "A Tavern in the Woods", "The Crumbling Keep").

`initializeLocationVoting`:
- Shuffles `AdventureMap.locations` randomly
- Selects N locations where N = player count
- Creates one `TextSubmission` per location in the LOCATION_VOTING phase

Each player independently votes for one location. Votes are individual — different players can
choose different locations and will end up on separate paths through the dungeon.

`handleLocationVoting` (fires on resolution):
- Reads each player's vote and sets `Player.selectedLocationId` to the `Location.id` they chose
- Immediately calls `handleLocationTraits`:
  - Each player receives all `Trait[]` defined on their chosen `Location` object
  - These are appended to `Player.traits` — the location visit is permanently recorded on the
    character before they've written or resolved a single encounter

---

## 5 — Encounter Writing (SET_ENCOUNTERS → WHAT_HAPPENS_HERE)

### SET_ENCOUNTERS

Players are assigned a location by join-order offset and name encounter labels for it. For
example, if assigned "A Tavern in the Woods", a player might write:
- "Drunk Bar Patrons"
- "A Strange Mapmaker"

These become `EncounterLabel` objects on `AdventureMap.encounterLabels[]`, tagged with the
`locationId` of the location they were written for.

### WHAT_HAPPENS_HERE

Players are assigned an `EncounterLabel` written by someone else (offset) and write the full
scene description (`Story.prompt`). The story is placed as an `Encounter` on the `GameBoard`
at coordinates `(currentX + i, y)` and carries `playerIds` — the list of players assigned to
this encounter (used to target repercussions later).

Players who selected the same location as the encounter's origin location will be the ones who
encounter it during MAKE_CHOICE_VOTING.

### WHAT_CAN_WE_TRY

Player is assigned 1–2 stories by offset and proposes action options for each.
Winning options become `Option` objects on the story.

### HOW_DOES_THIS_RESOLVE / HOW_DOES_THIS_RESOLVE_AGAIN

Player is assigned one story and writes the resolution for one of its options. This is where
**player class matters**: when making a text addition the player may attach a `Repercussion`
from the types their class unlocks:

```
TextAddition {
  authorId,
  addedText,
  repercussion: { repercussionType, repercussionSubmission }
}
```

Winning submissions have their repercussions collected and stored on `OutcomeFork.repercussions[]`.

---

## 6 — Individual Encounter Resolution (per player, per encounter)

After the writing phases conclude, each player takes a turn resolving the encounter that was
written for them. The encounter resolution loop runs one player at a time.

### MAKE_PARTNER_CHOICE_VOTING

Before their encounter begins, a player may choose to partner with another player. This is a
two-step mutual opt-in:

1. **MAKE_PARTNER_CHOICE_VOTING** — The active player votes for a partner from the available
   player list. The "winner" is whichever player received the most votes.
2. **ACCEPT_PARTNER_CHOICE_VOTING** — The chosen player votes to accept or decline.

`handleAcceptPartnerChoiceVoting`: If accepted, the partner's `authorId` is added to
`Story.partnerIds[]` on the active player's story. The relationship is mutual — the active
player's `authorId` is also added to the partner's story at the same location.

**Partners vote in both MAKE_CHOICE_VOTING and MAKE_OUTCOME_CHOICE_VOTING** alongside the
primary player. They also share in any repercussions that result from the encounter outcome.

### MAKE_CHOICE_VOTING

The active player (and any partners) vote on which `Option` to pursue.
→ `Story.selectedOptionId` is set to the winning vote.

### MAKE_OUTCOME_CHOICE_VOTING

If the chosen option has multiple `OutcomeFork` submissions, all other players vote on which
fork applies — deciding the active player's fate. If only one fork exists, it is selected
automatically.
→ `Option.selectedOutcomeForkId` is set; repercussion resolution fires immediately.

### Repercussion Resolution

`handleRepercussions` fires when `MAKE_OUTCOME_CHOICE_VOTING` resolves:

```
Target players = story.playerIds + story.partnerIds

TRAIT     → new Trait(repercussionSubmission, TraitType.STANDARD) appended to each target
TITLE     → new Trait(repercussionSubmission, TraitType.TITLE) appended — surfaces in displayName
COMPANION → new Trait(encounterLabel, TraitType.COMPANION) — the encounter itself becomes a
            permanent companion badge on the player's character sheet
ALL_PLAYERS → all of the above are applied to every player in the game instead of just targets

No repercussions + roundNumber < 2 → auto-generates a default "encounter again" SEQUEL effect
```

### NAVIGATE_WINNER

`PlayerCoordinates.x++`. Encounter resolution repeats for the next encounter in the round. When
all encounters are done:
- `roundNumber < 2` → increment round, loop back to WHAT_HAPPENS_HERE
- `roundNumber >= 2` → proceed to ENDING_PREAMBLE → WRITE_EPILOGUES

---

## 7 — Town Mode Stat Phases (Town Mode only)

Town Mode adds world-building phases that create shared stat types used in adventure resolution:

| Phase | Effect |
|---|---|
| `WHAT_DO_WE_FEAR` | Winning submission becomes a `StatType` (the fear entity) on `AdventureMap.statTypes` |
| `WHAT_ARE_WE_CAPABLE_OF` | Top 6 winning submissions become `StatType[]` on `AdventureMap.statTypes` |

The base player stats (strength, dexterity, charisma, intellect, wealth, magic, favor — all
initialized to 4) are checked against the relevant `StatType` to determine success or failure.

---

## 8 — Epilogue (WRITE_EPILOGUES)

Each player is assigned another player's character arc by join-order offset (`index - offset`).
They write how that character has changed over the adventure. The winning epilogue text is
attached to that player's record as an `Ending`.

---

## 9 — Final Player State

By the end of the game a player's record holds:

| Field | What it reflects |
|---|---|
| `playerClass` | The class chosen at creation — unchanged throughout |
| `traits[]` | Starting trait + location traits from LOCATION_VOTING + any TRAIT/COMPANION/TITLE repercussions earned during encounter resolution |
| `titles[]` | Separate list of title strings (mirrors TITLE-type traits - legacy) |
| `displayName` | `userName` + all TITLE traits joined with commas |
| `selectedLocationId` | The location they voted for in LOCATION_VOTING |
| `playerCoordinates` | Final position on the dungeon grid |
| Base stats | Initialized at 4; modified by adventure outcomes in Town Mode |
