OutcomeType Flow to Story Element Documentation (Streamlined Mode)

General Overview: The backend for nowhere is organized primarily in terms of text submissions and outcome types. 
Players submit TextAddition objects in CollaborativeTextHelper.submitTextAddition which are then converted into 
various story elements in the calculateWinningSubmission. Each time the game moves to another collaborative
writing phase, players are assigned text written by others which the frontend retrieves by utilizing
CollaborativeTextHelper.getOutcomeTypes. When an author submits a new TextAddition, it includes the outcomeType
they chose to indicate what their TextAddition was building upon.

Phase: LOCATION_VOTING  [locationVoting flag only]

Initialization (initializeLocationVoting):
- Called once on entering LOCATION_VOTING from PREAMBLE (NOT re-called in round 2).
- Randomly selects N locations (N = player count) from AdventureMap, excluding visited locations.
- Creates one TextSubmission per selected location in the LOCATION_VOTING phase.
- In round 2, clearStoryWritingPhases clears votes but preserves submissions so the same
  location pool is reused without re-initialization.

getOutcomeTypes: N/A - players vote directly on the pre-populated TextSubmissions

Handler: handleLocationVoting
- Reads each player's vote (submissionId = locationId they voted for).
- Sets Player.selectedLocationId from the winning vote.

Phase: SET_ENCOUNTERS

locationVoting flag OFF:
  getOutcomeTypes: Not applicable - no outcomeTypes used
  Handler: addAllSubmissionsToAdventureMap
  OutcomeType Mapping: N/A - submissions become EncounterLabel objects directly
  - submission.getCurrentText() → EncounterLabel.encounterLabel
  - Auto-generated UUID → EncounterLabel.encounterId

locationVoting flag ON (round 1 only — skipped entirely in round 2):
  getOutcomeTypes:
  - Reads TextSubmissions stored in the LOCATION_VOTING phase (the N randomly selected locations).
  - Assigns one location per player using offset: submissions[(playerIndex + 1) % N].
  - Player at index i writes encounter labels for the location at index (i+1) % N (never their own).
  - OutcomeType.id = Location.id (stored as TextSubmission.submissionId)
  - OutcomeType.label = Location.label (stored as TextSubmission.currentText)
  Handler: addAllSubmissionsToAdventureMap (same as non-voting path)

Phase: WHAT_HAPPENS_HERE

getOutcomeTypes (Round 1):
- Returns all EncounterLabel objects from AdventureMap
- OutcomeType.id = EncounterLabel.encounterId
- OutcomeType.label = EncounterLabel.encounterLabel

getOutcomeTypes (Round > 1):
- Returns filtered EncounterLabels (excluding already-used ones)
- OutcomeType.clarifier = assigned Story.storyId (for branching)

Phase: HOW_DOES_THIS_RESOLVE / HOW_DOES_THIS_RESOLVE_AGAIN

getOutcomeTypes:
- Returns single OutcomeType per player with subTypes array
- OutcomeType.id = Story.storyId (parent)
- OutcomeType.label = Story.prompt
- OutcomeType.clarifier = from distributed story (if set)
- OutcomeType.subTypes[] = WHAT_CAN_WE_TRY submissions related to story:
    - subType.id = TextSubmission.submissionId
    - subType.label = TextSubmission.currentText (the option text)

Handler: handleHowDoesThisResolve
